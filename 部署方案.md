下面这套方案适合你现在的目标：**把现有本地 AI 漫剧工作流，升级成“多人可登录、可后台管理、可限制模型额度、可部署到云服务器体验”的 Web 系统**。

我看了你的仓库，当前项目已经有本地网页控制台、CLI 工作流、硅基流动文本/图片/视频模型、腾讯云 TTS，以及 `.env` 密钥配置；网页端主要用于 AI 生成剧本、导入编辑、生成控制、查看日志和产物。([GitHub][1]) 目前 `manga_flow.web` 是基于 Python `ThreadingHTTPServer` 的本地单页控制台，任务日志写在 `outputs/web_jobs/` 和 `outputs/web_api/`。([GitHub][2]) 这非常适合本地开发，但 Python 官方也明确提示 `http.server` 不推荐用于生产环境，因为只做了基础安全检查。([Python documentation][3])

## 一、推荐总体架构

建议不要在现有 `web.py` 上硬塞登录和后台，而是把它升级成下面这种结构：

```text
用户浏览器
   ↓
Nginx 反向代理，负责 HTTPS / 静态资源 / 访问入口
   ↓
FastAPI 后端
   ├─ 登录认证 Auth
   ├─ 用户后台 Admin
   ├─ 额度系统 Quota
   ├─ 任务系统 Jobs
   ├─ 项目管理 Projects
   └─ 调用现有 manga_flow 工作流
          ↓
      模型供应商
      ├─ SiliconFlow 文本/图片/视频
      └─ Tencent TTS
```

选择 FastAPI 是比较自然的，因为你的项目已经是 Python，当前依赖里有 `httpx`、`pydantic`、`typer`、`jinja2` 等，但还没有正式 Web 框架、数据库 ORM 和认证库。([GitHub][4]) FastAPI 官方也支持 OAuth2 Password Bearer、JWT、密码哈希这类认证模式，适合你这种“自己发账号给用户登录”的场景。([FastAPI][5])

## 二、你最终要实现的核心能力

你的需求可以拆成 5 个系统：

| 模块   | 作用                     |
| ---- | ---------------------- |
| 账户认证 | 用户拿你给的账号密码登录           |
| 后台管理 | 管理员创建账号、改密码、禁用账号、设置额度  |
| 额度限制 | 每个账号能调用多少次 AI / 消耗多少点数 |

## 五、额度限制设计

这是你项目里最关键的部分。

### 方案 A：按“操作点数”计费，最适合第一版

先不要一开始就精确计算每个 token 或每秒视频成本。你可以定义一套内部点数：

| 操作      |                建议消耗 |
| ------- | ------------------: |
| AI 生成剧本 |                20 点 |
| 规范化导入剧本 |                 5 点 |
| 生成单张图片  |                10 点 |
| 生成单段配音  |                 3 点 |
| 生成单段视频  |                40 点 |
| 一键完整出片  | 按镜头数预估，比如 100～300 点 |

这样非常好管理，用户也容易理解。

### 方案 B：按真实模型成本计费，适合后期

如果硅基流动或其他接口返回 token、图片、视频成本，你可以把真实消耗写入 `usage_events`，再换算成额度。第一版不建议太复杂。

## 六、额度扣减流程

每次用户点击“AI 生成剧本 / 生成图片 / 一键出片”时，后端应该这样处理：

```text
1. 识别当前登录用户
2. 根据操作类型预估本次消耗
3. 检查用户剩余额度
4. 额度不足 → 拒绝任务
5. 额度足够 → 先预扣 reserved_quota
6. 创建 job
7. 后台执行模型调用
8. 成功 → reserved 转 used
9. 失败 → 退还未使用额度
10. 写 usage_events 流水
```

关键点：**要先预扣额度，再开始模型调用。**

否则用户可以同时开多个任务，把余额透支掉。

### 额度判断伪代码

```python
def reserve_quota(user_id: int, units: int):
    quota = get_quota_for_update(user_id)

    available = quota.monthly_quota - quota.used_quota - quota.reserved_quota

    if available < units:
        raise QuotaNotEnoughError("额度不足")

    quota.reserved_quota += units
    create_usage_event(
        user_id=user_id,
        estimated_units=units,
        status="reserved",
    )
```

这里需要数据库事务，避免两个并发请求同时通过额度检查。

## 八、用户隔离设计

你现在的产物默认在类似：

```text
outputs/<project_id>/episode_001/
```

多人系统里建议改成：

```text
outputs/users/<user_id>/<project_id>/episode_001/
```

项目 YAML 也建议从：

```text
data/projects/<name>.yaml
```

改成：

```text
data/users/<user_id>/projects/<name>.yaml
```

这样可以避免 A 用户看到 B 用户的剧本、日志和视频。

### 文件下载接口必须做权限检查

例如：

```text
GET /api/file?path=outputs/users/3/xxx/video.mp4
```

后端必须检查：

```text
当前登录用户 id == path 里的 user_id
或者当前用户是 admin
```

不能只靠隐藏前端链接。

## 十一、部署方案

### 第一版云服务器结构

```text
Ubuntu 云服务器
├─ Nginx：监听 80 / 443
├─ FastAPI + Uvicorn：监听 127.0.0.1:8000
├─ PostgreSQL 或 SQLite
├─ Redis，可选
├─ outputs/ 持久化目录
└─ .env 保存模型密钥
```

FastAPI 官方也有“Behind a Proxy”的部署说明，典型做法是让用户访问 Nginx，Nginx 再把请求转发给后端服务；后端服务只监听本机地址。([FastAPI][7])

### Nginx 代理建议

用户访问：

```text
http://服务器IP
```

Nginx 转发到：

```text
http://127.0.0.1:8000
```

生产环境建议最终加域名和 HTTPS。测试体验阶段可以先用 IP，但模型调用密钥一定不能暴露给前端。

### systemd 服务

建议创建：

```text
ai-manga.service
```

让后端挂掉后自动重启。

示意：

```ini
[Unit]
Description=AI Manga Workflow Web Server
After=network.target

[Service]
WorkingDirectory=/opt/ai_manga_workflow
EnvironmentFile=/opt/ai_manga_workflow/.env
ExecStart=/opt/ai_manga_workflow/.venv/bin/uvicorn manga_flow.server.app:app --host 127.0.0.1 --port 8000
Restart=always
User=www-data

[Install]
WantedBy=multi-user.target
```

## 十二、服务器安全建议

你这个项目涉及模型密钥和生成成本，安全要比普通网页更重要。

必须做：

```text
1. 模型 API Key 只放服务器 .env
2. 前端永远不出现模型 Key
3. 所有生成接口都要求登录
4. 所有文件访问都检查 user_id
5. 后台接口必须 require_admin
6. 额度检查必须放在模型调用之前
7. 云服务器只开放 80/443/SSH
8. SSH 禁止弱密码，最好用密钥登录
9. 定期备份数据库和 outputs
```

不要做：

```text
- 不要直接把现在的 8765 端口开放到公网
- 不要只用 Nginx Basic Auth 代替账户系统
- 不要让所有用户共用同一个 outputs 目录权限
- 不要在前端写死“剩余额度”
```

## 十四、我建议你的第一版额度规则

为了快速上线体验版，先这样定：

| 用户类型   |         每月额度 | 适合场景        |
| ------ | -----------: | ----------- |
| 测试账号   |        100 点 | 只能体验剧本和少量图片 |
| 普通体验账号 |        500 点 | 可以跑一两个短项目   |
| 高级体验账号 |       2000 点 | 可以完整跑多个项目   |
| 管理员    | 不限或 999999 点 | 你自己测试       |

消耗规则：

| 操作      |       扣点 |
| ------- | -------: |
| AI 生成剧本 |       20 |
| 导入剧本规范化 |        5 |
| 生成图片，每张 |       10 |
| 生成配音，每段 |        3 |
| 生成视频，每段 |       40 |
| 合成成片    |        5 |
| 一键完整出片  | 按镜头预估后预扣 |

一键完整出片预估：

```text
预估点数 = 剧本生成 0/20
        + 图片镜头数 × 10
        + 配音段数 × 3
        + 视频镜头数 × 40
        + 合成 5
```

## 十五、最终推荐结论

你的项目现在已经有完整的 AI 漫剧生产流程，真正缺的是“生产级 Web 壳子”。推荐路线是：

```text
保留现有 manga_flow 工作流
```
